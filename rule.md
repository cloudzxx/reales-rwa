# 合规检测对 RWA 应用流程带来的变化

## 核心变化：从「盲转」到「透明」

- **之前**：RWA 转账就是一笔普通 ERC20/SPL transfer，链上只有 hash 和 value，看不出风险
- **之后**：每次 transfer 可以触发合规检查，返回——这笔资金 30 分钟前来自一个混币器，3 天内交易 60 次，关联地址中有 OFAC 制裁地址

这是从**「转账状态」**到**「转账 + 画像」**的飞跃。

## 对 RWA 全生命周期的具体变化

| 阶段 | 之前 | 之后（有 6 层合规检测管道） |
|------|------|---------------------------|
| **发行前** | 合约参数设定 | 设定合规策略：白名单 + 规则阈值 + 自动冻结地址列表 |
| **申购** | 用户直接转 USDC，查不到背景 | 申购前调合规 API → 高风险地址冻结 / 拒绝准入 |
| **二级转账** | 任意白名单地址互转 | 每笔交易落地前触发规则引擎 + LLM 分析，自动拦截 mixer / sanctioned 交互 |
| **分红/回购** | 批量转账 | 转账中自动识别异常收款地址，触发人工审核 |
| **审计/监管** | 手动查区块浏览器 | 一键生成 `case_id` 报告：资金流图 + 4 维评分 + audit_trail，可直接用于监管报送 |

## 具体工作流变化

```
RWA 发行方视角

1. 创建 RWA 代币（指定 cap、白名单管理地址）
2. 用户发起 KYC → 加入白名单
3. 用户转账/申购 → 触发合规检查
   ├─ 规则引擎实时检测：大额/高频/混币器/制裁
   ├─ 地址画像识别对手方类型（exchange/mixer/EOA）
   ├─ 四维评分输出风险等级
   └─ LLM 生成叙事报告 + 建议动作
4. 高风险（critical/high）→ 自动冻结（freeze）
   中风险（medium）→ 触发人工审核（review）
   低风险（low）→ 放行并标记监控（monitor）
5. 生成合规报告：case_id + audit_trail → 归档
6. 监管查询 → 5 秒内调出历史案例
```

## 能力矩阵

| 能力 | 实现效果 |
|------|----------|
| **自动拦截** | 混币器地址转 RWA → 冻结，制裁地址交互 → 冻结 + 报告 |
| **风险评级** | 每个地址 / 每笔交易有 0-100 评分 + risk_level，可做分级 KYC |
| **合规追溯** | 回溯 50000 块的交易历史，递归 2 层对手方，生成完整 audit_trail |
| **AI 叙事** | 非技术人员读 LLM 报告即可理解风险（而非看原始 hex 数据） |
| **监管准备** | `case_id` + `generated_at` + `version` + `audit_trail` 符合金融监管报告格式规范 |

## 真实场景对比

> **一个 RWA 房地产代币，某用户要买 5000 USDC 份额**
>
> - **无合规**：合约检查白名单 → 通过 → 转账完成
> - **有合规**：查到该地址 24 小时前从 Tornado Cash 收到资金，规则引擎标记 `mixer_interaction(high)→freeze`，交易被冻结，合规官收到含完整审计链路的报告

## 一句话总结

合规检测把 RWA 从一个**开放的代币转账系统**变成了**带监管级风控的合规代币平台**——每笔交易都有自动化风险评估、可审计轨迹、和可读性分析报告，这是真实金融机构接受 RWA 的必要前提。
